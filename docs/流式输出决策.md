## 最终决策

**数据层分 5 类。UI 层只显示 4 种常驻块，加 1 种临时交互块。**

之前的 8 类太细了。
`Tool Call / Progress / Result` 不应拆成三类。它们本质上是同一个工具操作的生命周期。

# OMP Desktop 的 5 类数据

## 1. Narrative

Agent 产生的文字。

```ts
type NarrativeKind = 'reasoning' | 'intermediate' | 'final'
```

- `reasoning`：思考、计划。
- `intermediate`：工具调用前后的说明文字。
- `final`：整个 Run 的最终回答。

**只有 `final` 始终展开。**

Reasoning 和 Intermediate 合并进一个 `Process` 折叠块。
不要每段文字生成一个折叠卡片。

OMP RPC 的 `message_update` 已经包含 text、thinking、toolcall 增量。

---

## 2. Action

所有工具操作统一成一种类型：

```ts
type ActionKind = 'context' | 'command' | 'edit' | 'subagent' | 'external'
```

一个 Action 包含完整生命周期：

```text
Tool Call
→ Running
→ Progress
→ Result / Error
```

统一使用 `toolCallId` 更新同一个对象。

OMP 已经提供：

- `tool_execution_start`
- `tool_execution_update`
- `tool_execution_end`

不要把它们渲染成三个卡片。

### Context Action

以下工具连续出现时自动聚合：

```text
read
grep
glob
list
web search
```

显示为：

```text
Gathered context · 8 files · 3 searches        ▸
```

这直接采用 OpenCode 的做法。

---

## 3. Interaction

Agent 等待用户参与：

```ts
type InteractionKind = 'approval' | 'confirm' | 'question' | 'select' | 'input'
```

例如：

```text
Run command
npm install sharp

[Deny] [Allow once] [Always allow]
```

等待期间始终展开。
用户处理后，收回对应 Action 卡片。

OMP RPC 已有 `extension_ui_request`，支持 `confirm`、`select`、`input`、`editor` 等请求。

OMP 自身也已有：

- `read / write / exec` 工具等级。
- `allow / deny / prompt` 策略。
- `always-ask / write / yolo` 模式。

所以 Desktop 不需要新造审批协议。

---

## 4. Artifact

从 Action 结果中提取出的高价值产物：

```ts
type ArtifactKind = 'change-set' | 'file' | 'image' | 'report'
```

最重要的是 `change-set`，也就是 Diff。

### Diff 的归属

传输层：

```text
Edit Action
└─ Tool Result
   └─ details.diff
```

UI 层：

```text
Artifact: ChangeSet
```

OMP 的 Edit 结果已经提供：

- `details.diff`
- `firstChangedLine`
- `diagnostics`
- `perFileResults`

因此：

- 单次 Edit 卡片保存自己的 Diff。
- 整个 Run 再生成一个汇总 ChangeSet。
- 聊天流只显示摘要。
- 点击 `Review` 打开独立 Diff 面板。

```text
Changed 4 files · +82 −17                   [Review]
```

---

## 5. Lifecycle

控制整个 Run 的状态：

```ts
type RunState =
  | 'running'
  | 'waiting'
  | 'completed'
  | 'interrupted'
  | 'failed'
  | 'retrying'
  | 'compacting'
```

它不是聊天卡片。

它只控制：

- 顶部状态。
- Spinner。
- Stop 按钮。
- 等待审批提示。
- 完成时间。
- 错误状态。

---

# 最终 UI 只保留这几块

```text
用户消息

Completed · 16 actions · 32s                     ▸
Changed 4 files · +82 −17                  [Review]

最终回答始终展开
```

运行过程中可能临时出现：

```text
Waiting for approval

Run npm install sharp
[Deny] [Allow once] [Always allow]
```

# 最终结构

```text
Agent Run
├─ Narrative
│  ├─ Reasoning
│  ├─ Intermediate
│  └─ Final
├─ Action
│  ├─ Context
│  ├─ Command
│  ├─ Edit
│  ├─ Subagent
│  └─ External
├─ Interaction
├─ Artifact
└─ Lifecycle
```

## 最关键的三个决定

1. **Final 不是独立大类，只是 Narrative 的最终状态。**
2. **Tool Call、Progress、Result 合并成一个 Action。**
3. **Diff 来源于工具结果，但产品层必须升级成 Artifact。**

这套分类最适合 OMP RPC。转换层很薄，也不会重复造轮子。
