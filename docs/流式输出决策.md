# 流式输出展示模型

> 本文解释流式输出展示模型的设计依据。正式约束以
> `decision/04-streaming-conversation-and-run-trace.md`、`docs/product-spec.md`
> 和对应 TODO 为准。

## 结论

**Renderer 的 Run 展示投影分 5 类。最终回答和 Process 摘要常驻，Interaction 只在等待用户处理时展开。**

之前的 8 类太细了。
`Tool Call / Progress / Result` 不应拆成三类。它们本质上是同一个工具操作的生命周期。

---

## 现有项目的做法

| 项目     | 核心做法                                                                                                 | 我们借鉴什么                      |
| -------- | -------------------------------------------------------------------------------------------------------- | --------------------------------- |
| Goose    | Thinking、普通文本、工具卡片。工具调用和结果在同一卡片中，审批也放在卡片内。                             | 工具操作统一成一个 Action         |
| OpenCode | 基础 Part 是 Text、Reasoning、Tool。连续的 Read、Glob、Grep、List 会聚合成一个 Context Group。           | 低价值工具聚合，不逐条占空间      |
| Zed      | 审批显示在对应工具卡片中。文件变化单独进入 Review Changes，可以按 Hunk 接受或拒绝。([Zed][1]) ([Zed][2]) | Approval 和 Diff 都不能当普通日志 |

# OMP Desktop 的 5 类数据

## 1. Narrative

Agent 产生的文字。

```ts
type NarrativeKind = 'reasoning' | 'intermediate' | 'final'
```

- `reasoning`：思考、计划。
- `intermediate`：工具调用前后的说明文字。
- `final`：整个 Run 的最终回答。

**只有 `final` 始终展开。**

Reasoning 和 Intermediate 合并进一个 `Process` 折叠块。
不要每段文字生成一个折叠卡片。

OMP RPC 的 `message_update` 已经包含 text、thinking、toolcall 增量。

---

## 2. Action

所有工具操作统一成一种类型：

```ts
type ActionKind = 'context' | 'command' | 'edit' | 'subagent' | 'external'
```

一个 Action 包含完整生命周期：

```text
Tool Call
→ Running
→ Progress
→ Result / Error
```

统一使用 `toolCallId` 更新同一个对象。

OMP 已经提供：

- `tool_execution_start`
- `tool_execution_update`
- `tool_execution_end`

不要把它们渲染成三个卡片。

### Context Action

以下工具连续出现时自动聚合：

```text
read
grep
glob
list
web search
```

显示为：

```text
Gathered context · 8 files · 3 searches        ▸
```

这直接采用 OpenCode 的做法。

---

## 3. Interaction

Agent 等待用户参与：

```ts
type InteractionKind = 'approval' | 'confirm' | 'question' | 'select' | 'input'
```

例如：

```text
Run command
npm install sharp

[Deny] [Approve]
```

等待期间始终展开。
用户处理后，收起 Interaction 控件，保留对应 Action 的最终状态和历史。

OMP RPC 已有 `extension_ui_request`，支持 `confirm`、`select`、`input`、`editor` 等请求。

OMP 自身也已有：

- `read / write / exec` 工具等级。
- `allow / deny / prompt` 策略。
- `always-ask / write / yolo` 模式。

所以 Desktop 不需要新造审批协议。

---

## 4. Artifact

从 Action 结果中提取出的高价值产物：

```ts
type ArtifactKind = 'change-set' | 'file' | 'image' | 'report'
```

最重要的是 `change-set`，也就是 Diff。

### Diff 的归属

传输层：

```text
Edit Action
└─ Tool Result
   └─ details.diff
```

UI 层：

```text
Artifact: ChangeSet
```

OMP 的 Edit 结果已经提供：

- `details.diff`
- `firstChangedLine`
- `diagnostics`
- `perFileResults`

MVP 只显示 RPC 原始结果能够可靠提供的改动摘要，不在对话流展开完整 Diff。

POST-MVP 的 Changes / Review / Diff 再负责：

- 从单次 Edit 结果提取 Diff。
- 聚合整个 Run 的最终 ChangeSet。
- 处理同一文件的多次编辑。
- 检测 Bash、Write 等其他工具造成的工作区变化。
- 点击 `Review` 打开独立 Diff 面板。

```text
Changed 4 files
```

---

## 5. Lifecycle

控制整个 Run 的状态：

```ts
type RunState =
  | 'running'
  | 'waiting'
  | 'completed'
  | 'interrupted'
  | 'failed'
  | 'retrying'
  | 'compacting'
```

它不是聊天卡片。

它只控制：

- 顶部状态。
- Spinner。
- Stop 按钮。
- 等待审批提示。
- 完成时间。
- 错误状态。

---

# 最终 UI 只保留这几块

```text
用户消息

Completed · 16 actions · 32s                     ▸
Changed 4 files

最终回答始终展开
```

运行过程中可能临时出现：

```text
Waiting for approval

Run npm install sharp
[Deny] [Approve]
```

# 最终结构

```text
Agent Run
├─ Narrative
│  ├─ Reasoning
│  ├─ Intermediate
│  └─ Final
├─ Action
│  ├─ Context
│  ├─ Command
│  ├─ Edit
│  ├─ Subagent
│  └─ External
├─ Interaction
├─ Artifact
└─ Lifecycle
```

## 最关键的三个决定

1. **Final 不是独立大类，只是 Narrative 的最终状态。**
2. **Tool Call、Progress、Result 合并成一个 Action。**
3. **Diff 来源于工具结果；MVP 只显示摘要，Run 级 ChangeSet 和 Review 在 POST-MVP 实现。**

这套分类最适合 OMP RPC。转换层很薄，也不会重复造轮子。

[1]: https://zed.dev/docs/ai/tool-permissions 'Tool Permissions | Zed Code Editor Documentation'
[2]: https://zed.dev/docs/ai/agent-panel 'Agent Panel | AI Coding Agent - Zed Agent Panel'

对。**好执行的核心原因，就是 OMP RPC 已经覆盖了大部分底层事件。**

但不是 5 类都能直接一一收到。准确说：

| UI 分类     | RPC 支持情况 |
| ----------- | ------------ |
| Narrative   | 基本直接支持 |
| Action      | 直接支持     |
| Interaction | 基本直接支持 |
| Artifact    | 部分直接支持 |
| Lifecycle   | 直接支持     |

### 1. Narrative

RPC 的 `message_update` 能收到：

- `text_delta`
- `thinking_delta`
- `toolcall_delta`

所以普通文本、思考、工具调用内容都能流式收到。

但 `final` 不是 RPC 明确标记的。

需要 Desktop 判断：

```text
最后一段 Assistant Text
+ 后面没有工具调用
+ 收到 agent_end
= Final
```

---

### 2. Action

这是支持最完整的：

```text
tool_execution_start
tool_execution_update
tool_execution_end
```

三种事件都有 `toolCallId`，可以更新同一张工具卡片。

所以不需要自己猜：

- 工具什么时候开始。
- 工具是否还在运行。
- 工具返回了什么。
- 是否失败。

---

### 3. Interaction

RPC 已经有：

```text
extension_ui_request
extension_ui_response
```

支持：

- confirm
- select
- input
- editor
- cancel

所以审批、提问、选择都可以直接做。

OMP v17.0.6 的通用工具审批实际通过 `extension_ui_request: select` 提供 `Approve` 和 `Deny`。其他交互仍按 RPC 实际提供的 method 和选项渲染，Desktop 不自行扩展授权范围。

---

### 4. Artifact

这部分是“部分直接支持”。

例如 Edit 工具直接返回：

```text
result.details.diff
result.details.diagnostics
result.details.perFileResults
```

因此单次 Edit Diff 可以直接获取。

但下面这些需要 Desktop 派生：

- 整个 Run 的最终 ChangeSet。
- 多次编辑同一个文件后的最终 Diff。
- Bash 命令造成的文件变化。
- Write 工具造成的前后 Diff。

所以 Artifact 是：

```text
RPC 原始结果
+ Desktop 聚合
```

---

### 5. Lifecycle

RPC 直接提供：

```text
agent_start
agent_end
turn_start
turn_end
auto_compaction_start
auto_compaction_end
auto_retry_start
auto_retry_end
```

可以直接控制运行状态。

---

## 最准确的说法

不是：

> RPC 直接输出了我们定义的 5 类。

而是：

> RPC 已经输出了足够完整的原始事件，Desktop 只需要做一层很薄的归一化。

最主要的推断只有三个：

1. 判断哪段文本是 Final。
2. 聚合 Read、Grep、Glob 等 Context Action。
3. POST-MVP 生成整个 Run 的最终 ChangeSet。

所以这套分类不是凭空设计的。它和 OMP RPC 的能力是基本对齐的。
