## 结论

这块必须现在做。它属于 Desktop 的 **P0 基础能力**。

但不要自己实现 OAuth、Token 刷新和模型发现。OMP 已经完成了大部分。Desktop 只需要复用 RPC，再补少量接口。

## OMP 的整体结构

```text
ProviderDefinition
        ↓
AuthStorage
        ↓
ModelRegistry
        ↓
AgentSession
        ↓
RPC
        ↓
Electron Main → Preload → Renderer
```

### 1. ProviderDefinition：描述登录方式

每个 Provider 使用统一定义：

- Provider ID、名称
- 登录函数
- Token 刷新函数
- OAuth 凭证转 API Key 的函数
- 回调端口
- 是否需要用户粘贴授权码
- 凭证实际存储到哪个 Provider ID

因此，新增 Provider 基本就是增加一个定义文件，而不是修改一堆分支。fileciteturn7file0L23-L56

OMP 还把两类登录统一了：

- OAuth 登录返回 `OAuthCredentials`
- API Key 登录直接返回字符串

很多 Provider 的“登录”，本质上就是打开 Key 页面，让用户粘贴 API Key，再进行验证。fileciteturn31file0L3-L8 fileciteturn31file0L55-L115

## 授权和凭证怎么管理

核心类是 `AuthStorage`。

它管理：

- API Key
- OAuth Access Token
- Refresh Token
- Token 自动刷新
- 多账号
- 账号 ID、邮箱、组织信息
- 限流状态
- 凭证轮换
- Session 级凭证粘性

登录完成后，凭证统一写入 `AuthStorage`。OAuth 和 API Key 走同一个入口。fileciteturn9file1L2604-L2669

默认持久化是 SQLite。数据库文件权限在 Unix 上设置为 `0600`，目录设置为 `0700`。它还使用 WAL、锁等待和 Refresh Lease，处理多个 OMP 进程同时刷新 Token 的问题。fileciteturn9file1L6075-L6081 fileciteturn9file1L6234-L6276

实际调用模型时，凭证优先级是：

1. CLI Runtime Override
2. `models.yml` Provider API Key
3. OAuth Token，自动刷新
4. 通过登录保存的 API Key
5. 环境变量
6. 其他持久化 API Key
7. 自定义 fallback resolver

fileciteturn9file1L4792-L4856

所以 Desktop 不应该读取 SQLite，也不应该自己刷新 Token。只调用 `AuthStorage` 所在的 Runtime。

## 模型列表怎么获取和选择

核心类是 `ModelRegistry`。

模型来源包括：

- OMP 内置模型目录
- SQLite 模型缓存
- Provider 动态模型发现
- `models.yml`
- Extension 动态注册

登录成功后，RPC 会调用：

```ts
modelRegistry.refreshProvider(providerId, 'online')
```

这会重新请求该 Provider 的模型目录，不继续使用旧缓存。fileciteturn15file0L132-L179 fileciteturn26file0L137-L141

`getAvailableModels()` 只返回已经有授权的 Provider 模型。判断本身是同步的，不会在列表页面触发 OAuth 刷新。fileciteturn17file0L254-L289

模型选择使用：

```text
provider + modelId
```

不是只使用模型名称。fileciteturn18file0L38-L70

## 推理强度怎么获取

不要硬编码“低、中、高”。

每个模型对象已经包含：

```ts
model.reasoning
model.thinking
```

其中：

```ts
model.thinking.efforts
model.thinking.defaultLevel
model.thinking.mode
model.thinking.effortMap
```

标准强度全集是：

```text
minimal
low
medium
high
xhigh
max
```

fileciteturn29file0L27-L87 fileciteturn30file0L3-L20

Desktop 逻辑应该是：

```ts
if (!model.reasoning) {
  // 不显示推理强度
} else if (!model.thinking) {
  // 模型会推理，但用户不能控制强度
} else {
  // 下拉框直接显示 model.thinking.efforts
}
```

不同模型支持的档位不同。例如有的只有 `low/high`，有的支持 `low/medium/high/xhigh/max`。这些信息在模型构建时已经计算完成。fileciteturn27file0L54-L96 fileciteturn27file0L124-L153

## 当前 RPC 已经支持什么

现在已有：

```text
get_login_providers
login
get_available_models
set_model
set_thinking_level
cycle_thinking_level
get_state
```

fileciteturn21file0L29-L89

登录流程已经适合 Electron：

1. Desktop 调用 `login(providerId)`。
2. RPC 发出 `open_url` UI 请求。
3. Electron Main 使用 `shell.openExternal()`。
4. 需要粘贴授权码或 API Key 时，RPC 发出 `input` 请求。
5. Desktop 返回用户输入。
6. Runtime 保存凭证。
7. Runtime 刷新模型列表。
8. RPC 返回登录成功。

现有 `RpcClient.login()` 已经封装了 `onOpenUrl` 和 `onManualCodeInput`。fileciteturn35file0L157-L220

RPC 服务端也明确实现了这个流程。fileciteturn26file0L87-L145

## 当前 RPC 的缺口

现有 RPC 足够做基础登录，但不足以做完整的账号管理页面。

至少缺少：

- `logout`
- 查看登录账号列表
- 删除指定账号
- 查看凭证类型：OAuth 或 API Key
- 查看邮箱、组织、账号 ID
- 检查凭证健康状态
- 手动刷新模型目录
- 查看授权来源：环境变量、OAuth、配置文件等

还有一个明确问题：

`get_login_providers` 使用 `provider.id` 判断是否已登录，但某些登录项会把凭证存储到另一个 Provider ID。

例如：

```text
openai-codex-device
    ↓ storeCredentialsAs
openai-codex
```

fileciteturn38file0L6-L20

RPC 当前没有返回 `storeCredentialsAs`，并且直接执行：

```ts
hasAuth(provider.id)
```

因此设备登录成功后，登录列表存在显示成“未登录”的风险。fileciteturn33file0L20-L27 fileciteturn26file0L87-L94

这个应优先修复。

## Desktop 的最小实现

Electron Main 持有 `RpcClient`。Renderer 不直接访问 OMP 进程。

Preload 只暴露这些接口：

```ts
auth.listProviders()
auth.login(providerId)
auth.logout(providerId)

models.list()
models.select(provider, modelId)

thinking.set(level)
session.getState()
```

登录页面流程：

```text
选择 Provider
→ 登录
→ 打开浏览器
→ 等待回调或输入 Key
→ 自动刷新模型
→ 选择模型
→ 根据 model.thinking.efforts 显示推理强度
```

API Key 不需要单独设计一套系统。继续复用 OMP 的统一 `login()` 流程。

**明确判断：**

- 不要在 Electron 中实现 OAuth。

- 不要直接操作 OMP SQLite。

- 不要维护自己的模型列表。

- 不要硬编码推理强度。

- 只补齐 RPC 的账号管理接口。

- Auth + Model Selector 应放在聊天功能之前完成。

**核心管理逻辑放在 Electron Main。**

四层职责这样划分：

| 层       | 职责                                                         |
| -------- | ------------------------------------------------------------ |
| Renderer | 展示 Provider、模型、推理强度。接收用户操作。                |
| Preload  | 提供受控、类型化 IPC 接口。不要写业务逻辑。                  |
| Main     | 管理 Runtime 进程、RPC、登录流程、模型状态和错误处理。       |
| Runtime  | 真正保存凭证、刷新 Token、发现模型、执行模型调用。保持不变。 |

## 推荐结构

```text
Renderer
  ↓ window.omp.auth.login()
Preload
  ↓ ipcRenderer.invoke()
Electron Main
  ↓ RpcClient.login()
Runtime RPC
```

Main 内部可以分三个 Manager：

```text
RuntimeManager
├── AuthManager
├── ModelManager
└── SessionManager
```

### AuthManager

负责：

- 获取 Provider 列表
- 发起登录
- 接收 `open_url`
- 调用 `shell.openExternal()`
- 接收用户粘贴的 API Key 或授权码
- 登录状态同步

不要保存 Token。Token 仍由 Runtime 保存。

### ModelManager

负责：

- 调用 `get_available_models`
- 登录成功后重新获取模型
- 调用 `set_model`
- 缓存模型的非敏感元数据
- 根据 `model.thinking.efforts` 提供推理强度列表

### Renderer

Renderer 只保存界面状态：

```ts
{
  ;(providers, models, selectedModel, selectedThinkingLevel, loginDialogState)
}
```

不要让 Renderer：

- 直接启动 Runtime
- 直接访问 RPC
- 直接读取数据库
- 持有 API Key
- 实现 OAuth

## Preload 接口

保持很薄：

```ts
window.omp = {
  auth: {
    listProviders(),
    login(providerId, inputHandler),
  },
  models: {
    list(),
    select(provider, modelId),
  },
  thinking: {
    set(level),
  },
  session: {
    getState(),
  },
}
```

## Runtime 不修改的限制

现有 RPC 已经能实现：

- 登录
- Provider 列表
- 模型列表
- 模型选择
- 推理强度选择

但完整账号管理暂时做不了，例如：

- 退出登录
- 删除指定账号
- 查看多个账号
- 查看凭证详情

这些 RPC 当前没有暴露。

因此 MVP 应只做：

```text
登录 → 选择模型 → 选择推理强度 → 开始对话
```

不要在 Main 层自己实现另一套授权存储。那会和 Runtime 状态分裂。

**最终判断：**

```text
Renderer：界面
Preload：桥
Main：管理中心
Runtime：能力和数据源
```

授权和模型管理的编排放在 **Main**。真实授权数据仍归 **Runtime**。
