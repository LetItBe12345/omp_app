# Workspace 工具权限

> 本文解释 OMP Desktop 的工具权限边界和实现依据。正式产品规则以
> `docs/product-spec.md` 为准，已确认决策见
> `decision/04-streaming-conversation-and-run-trace.md`。

## 结论

OMP 已经提供工具审批系统。Desktop 不重新判断工具是否危险，也不重新实现权限引擎。

OMP 将工具能力分成三个等级：

- `read`：读取。
- `write`：修改文件或会话。
- `exec`：执行 Shell、浏览器或子 Agent 等高权限操作。

OMP v17.0.6 提供三种审批模式：

| 模式         | 自动允许                 |
| ------------ | ------------------------ |
| `always-ask` | 读取；写入和执行仍需确认 |
| `write`      | 读取和写入；执行仍需确认 |
| `yolo`       | 读取、写入和执行         |

Desktop 默认使用 `write`。文件编辑不中断，执行类操作仍需用户确认。

## 单次审批

工具需要用户确认时，OMP 通过 RPC 发出 `extension_ui_request`。v17.0.6 的通用工具审批使用 `select`，实际选项为 `Approve` 和 `Deny`。

示例请求：

```json
{
  "type": "extension_ui_request",
  "method": "select",
  "id": "request-id",
  "title": "Allow tool: bash",
  "options": ["Approve", "Deny"]
}
```

Desktop 使用现有 `respondExtensionUi` 接口返回选择：

```ts
window.desktop.respondExtensionUi(requestId, {
  value: 'Approve'
})
```

每个需要审批的 Tool Call 都有独立请求。一次用户 Prompt 可能产生零个、一个或多个审批请求。

Desktop 的 UI 必须遵守以下规则：

- 审批控件显示在对应 Action 的原始位置，不打开居中模态框。
- 等待期间保持展开，并滚动到该位置。
- 只显示 RPC 实际提供的选项。
- 处理后收起 Interaction 控件，保留 Action 的最终状态和历史。
- 应用退出、Runtime 重启、请求超时或用户取消时，明确返回取消，不能让 OMP 永久等待。

## Workspace 审批模式

当前 OMP RPC 没有读取或修改审批模式的命令，`get_state` 也不返回该值。因此 Desktop 负责保存每个 Workspace 选择的模式，并在启动 OMP 时显式传入：

```text
omp --mode rpc --approval-mode <mode>
```

保存规则：

- 以规范化后的 Workspace 绝对路径作为隔离键。
- 保存到 Electron `userData` 下的 Desktop 配置。
- 不修改项目中的 `.omp/config.yml`。
- Workspace 没有保存值时使用 `write`。
- Renderer 显示 Desktop 为当前 Runtime 选择的模式，不能伪装成从 OMP 查询到的状态。

修改审批模式需要重启当前 OMP Runtime，并恢复原 Session。以下状态禁止切换：

- Agent 正在执行。
- Follow-up 队列非空。
- 正在等待 Permission。

切换到 `yolo` 前必须显示风险确认。用户取消后不修改配置，也不重启 Runtime。

## MVP 边界

MVP 只提供三种 Workspace 审批模式：

- `always-ask`
- `write`
- `yolo`

MVP 不提供：

- 每个工具的 `allow`、`prompt`、`deny` 编辑器。
- “当前 Session 始终允许”或“当前 Workspace 始终允许此工具”等 RPC 未提供的审批范围。
- Desktop 自己维护一套工具危险等级。
- 通过项目内配置文件保存 Desktop 的选择。

单工具规则以后可以作为高级设置单独设计，不能混入 MVP。

## 权限不等于沙箱

OMP 的权限系统是工具审批系统，不是操作系统级沙箱。

OMP Runtime 仍然是本机进程。只设置 Workspace `cwd` 不会限制它访问：

- Workspace 外的文件。
- 网络。
- 继承的环境变量。
- 用户账户有权访问的系统资源。

因此产品文案使用“权限”或“Workspace 工具权限”，不能使用“工作区隔离”。

如果以后需要真正的 Linux 进程隔离，应单独评估 Bubblewrap、容器或其他系统级方案，并处理 Git、Provider 登录、系统工具和桌面集成兼容性。这不属于 MVP。

## 组件职责

```text
Renderer
  ├─ 显示 Workspace 审批模式
  └─ 在对应 Action 中显示单次审批
          ↓
Preload
  └─ 暴露受控的权限模式和 respondExtensionUi 接口
          ↓
Main
  ├─ 按 Workspace 保存审批模式
  ├─ 管理未完成的 Extension UI 请求
  └─ 重启 Runtime 并恢复 Session
          ↓
OMP Runtime
  ├─ --approval-mode
  ├─ 原生工具等级和审批判断
  └─ extension_ui_request / response
```

Main 和 Renderer 只负责配置、展示和桥接。是否允许工具执行的最终判断仍由 OMP Runtime 完成。
