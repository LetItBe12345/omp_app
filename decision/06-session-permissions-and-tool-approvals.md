# 决策记录 06：Session 权限与工具审批

- 对应任务：`TODO/in-progress/06-session-permissions-and-tool-approvals.md`
- 确认日期：2026-07-23
- 适用版本：OMP v17.0.6

## Session 权限

1. 输入框下方的控制栏增加权限按钮，位置紧邻模型和 Thinking Level。按钮直接显示当前 Session 的短中文权限值：`权限：严格`、`权限：标准` 或 `权限：全部允许`。点击按钮打开一个小型选择框，框内只显示这三个中文名称，不显示说明或英文原值；选择后立即关闭，不增加“保存”按钮。已有 Session 立即开始切换；新建且尚未发送第一条消息的 Session 立即保存选择，但等首次发送时再启动 OMP。权限不设置全局快捷键，但按钮和选择框必须支持 Tab 聚焦、Enter、方向键和 Esc。
2. 权限按钮管理当前 Session 的 OMP 工具审批模式，不表示 Linux 文件系统沙箱，也不表示 Electron 系统权限。
3. Desktop 支持 OMP v17.0.6 的三种模式：`严格` 对应 `always-ask`，自动允许读取，写入和执行需确认；`标准` 对应 `write`，自动允许读取和写入，执行需确认；`全部允许` 对应 `yolo`，自动允许读取、写入和执行。主界面和权限选择框使用短中文名称，启动参数、日志和错误详情使用英文原值。即使当前是`全部允许`，OMP 发出的安全确认或其他 `select` 交互也必须照常显示，Desktop 不能按当前模式隐藏或直接跳过请求。
4. 新建 Session 的权限默认显示并保存为`全部允许`，用户可以在发送第一条消息前修改。发送第一条消息时，Main 用已保存的模式启动 OMP；配置按 Workspace ID 和 Session ID 写入 Desktop 数据目录，使用原子替换，不弹出单独的权限选择窗口，也不修改项目内的 `.omp/config.yml`。权限缺失或不是三个有效值之一时按 `yolo` 打开，记录诊断日志并补存正确值。Session 删除成功后清理权限、文字草稿、附件和历史缓存；删除失败时全部保留。归档不删除权限和文字草稿，但可以释放附件和历史缓存。文字草稿继续使用现有按 Workspace 和 Session 区分的 `localStorage`，不增加数据库；附件只在当前 Desktop 运行期间按 Session 保留，关闭应用后不恢复。
5. 新建、打开、切换或异常恢复当前 Session 时，Main 都读取该 Session 保存的模式，并通过固定枚举映射生成 `always-ask`、`write` 或 `yolo`，使用参数数组显式传入 `--approval-mode <mode>`，不经过 Shell，也不接受任意配置字符串。当前 RPC 没有读取或修改审批模式的命令，`get_state` 也不返回该值，因此 Renderer 显示 Desktop 为当前 Session 保存的值，不能退回默认值或沿用其他 Session 的权限。
6. MVP 的多个 Session 共用一个 OMP Runtime，`--approval-mode` 是进程级参数。目标 Session 与当前 Runtime 的模式不同时，Main 必须重启 Runtime，再恢复目标 Session；模式相同时可以复用当前 Runtime，权限按钮不显示转圈。重启期间不弹窗，不遮挡聊天页面，输入框保持可编辑；权限按钮显示小转圈，发送、权限修改和 Session 切换暂时禁用，不提供取消按钮。Renderer 只保留当前 Session 和最近离开的 1 个 Session 的历史展示投影；非当前缓存不含 RPC 原始事件、Tool 完整原始输出、图片 Base64、二进制内容、临时动画或进度，也不跨 Desktop 重启保存。目标 Session 有缓存时立即显示，OMP 恢复后仍以其返回的历史为准并按消息 ID 校正；没有缓存时等待 OMP 恢复，不为此复制聊天数据库。加载超过 150 毫秒后才在聊天区域中央显示浅色“正在打开会话…”，快速完成时不显示加载状态。输入草稿和附件必须保留。用户仍可查看 Session 列表，也可以直接关闭 Desktop，未成功的权限修改不保存。
7. 当前 Session 没有运行任务时允许修改权限，修改后重启 Runtime 并恢复该 Session。新 Runtime 就绪后才保存新权限；启动失败时恢复原权限，并尝试按原权限重新启动。恢复成功时在输入框附近提示“权限切换失败，已恢复为「当前权限」”，不进入整页错误状态。Session 打开失败时继续停留在目标 Session，权限按钮显示目标 Session 保存的值，并允许用户选择另一权限再次启动；新权限仍然只有在启动成功后才保存。已有历史缓存时在输入框附近显示错误和“重试”，没有缓存时把聊天区域的加载状态替换为“会话打开失败”和“重试”，不重复显示同一错误。新旧权限均启动失败时保留聊天记录、草稿和附件，显示 Runtime 错误以及“重试”和“查看错误”，重试仍使用原权限。启动等待沿用 Runtime Supervisor 的 15 秒超时，Session 切换失败后不自动重试。错误详情显示退出码和脱敏后的最后 20 KB `stderr`，支持复制，不显示 Runtime 环境变量、模型密钥或代理密码。切换到 `yolo` 不显示风险确认。Agent 正在执行、Follow-up 队列非空或正在等待 Permission 时禁用权限修改和 Session 切换；用户需要先 Stop。

## 工具审批

8. OMP 要求审批时，通过 RPC `extension_ui_request` 的 `select` 请求接收选项，并使用现有 `respondExtensionUi` 返回结果。Main 只有在 Runtime 版本为 OMP v17.0.6、`method` 精确为 `select`、`options` 精确为 `["Approve", "Deny"]` 且 `title` 第一行以`Allow tool: `开头时，才按工具权限处理；不做模糊匹配。未知 Runtime 版本或格式匹配失败时回退为通用 Interaction，原样显示标题和选项，不进入权限合并区，也不执行 Desktop 的自动选择；OMP 提供超时时间时显示剩余时间，到期后移除按钮并显示“已超时”，没有超时时一直等待选择或取消。每次未知版本 Runtime 启动最多显示一次“当前 OMP 版本未验证，权限确认使用兼容模式”，同时记录格式不兼容的诊断日志。工具权限按钮显示为“允许”和“拒绝”，回传值仍使用英文原值。单个权限请求的按钮顺序为“拒绝、允许”，默认焦点放在“允许”；Enter 执行当前焦点，Esc 回传 `Deny`。工具审批默认等待 30 秒；OMP 请求携带更短超时时间时采用较短值，携带更长时间时仍采用 30 秒。Main 保存绝对截止时间，Renderer 显示实际剩余秒数；窗口失焦、最小化或系统睡眠不重置期限，唤醒时已经过期则立即自动回传 `Approve`。该规则使`严格`和`标准`成为限时确认，而不是无限期阻断。Desktop 不自行增加授权范围。
9. 同一 Run 有多个工具权限请求时，Main 并行接收，并在第一个请求到达后等待 100 毫秒收集同批请求；Renderer 把它们合并到一个确认区，不使用多个独立卡片，也不隐藏为串行队列。只有一个请求时不显示计数和批量按钮；追加第二个请求时原区域变成合并确认区，保持原焦点、截止时间和页面位置。合并区显示`待确认 N / 总数`，每行可以单独拒绝或允许，底部固定“全部拒绝”和“全部允许”；默认焦点放在“全部允许”，Esc 表示全部拒绝。Main 从审批 `title` 提取中文工具类别和主要目标，只把紧凑摘要、请求 ID、状态和截止时间发给 Renderer；完整 `title` 只在 Main 内存中保留，不进入 Renderer、不写日志、不持久化。每行例如`命令 · npm install`、`写入 · src/app.ts`，不显示写入 Content；未知工具保留原始名称。Workspace 内路径显示相对路径，外部路径显示绝对路径；URL 隐藏协议、用户信息和查询参数，只显示域名与路径；命令在 Workspace 子目录或外部执行时，在命令前显示相对或绝对工作目录；多文件请求显示第一个路径和文件总数。无法可靠处理时只显示原始工具名称。摘要始终只占一行，按确认区实际可用宽度显示，超出部分使用省略号，不设置固定字符上限。标题解析只生成显示摘要，不能用于关联 Tool Call。确认区最多直接显示 5 行，更多内容在区域内滚动。新请求追加到末尾，不抢焦点、不强制滚动，并沿用当前确认区的剩余时间；用户处理单项和新请求追加都不重置截止时间。OMP 自带更短期限时采用能够保证请求有效的较短期限；到期时只自动允许尚未处理的请求。
10. 合并确认区放在当前 Run 的过程区域底部。请求到达时自动展开过程并滚动到确认区；用户可以再次折叠，折叠摘要显示`等待操作 · N 秒后自动允许`，倒计时继续。折叠状态下自动允许后不重新展开，摘要恢复为“进行中”；新的权限请求到达时再次自动展开。用户单独处理一项后，Main 立即向 OMP 回传，不等待整组完成。点击“全部允许”或“全部拒绝”只处理点击瞬间已经进入确认区的未处理请求，之后到达的请求必须重新显示。已处理的行不立即删除，改为“已允许”“已自动允许”“已拒绝”“已取消”或“请求已失效”，避免列表跳动。OMP `cancel` 到达时停止对应倒计时并禁用该行，不能再自动允许。相同请求 ID 的重复事件只保留第一条，不增加行、不重置倒计时、不重复回传，并记录诊断警告。请求已经完成或取消后，Main 忽略 Renderer 的迟到操作，用最终状态纠正界面，不弹错误。响应时 OMP 已断开则标记“请求已失效”，不向新 Runtime 重放。用户 Stop 时取消全部待处理请求和倒计时，不允许任何剩余请求；关闭 Desktop 时直接结束 Runtime，不等待倒计时，也不保存或重放待处理请求。全部请求完成或取消后确认区立即消失，结果保留到对应 Tool Call 状态。每个请求只能完成一次；用户操作与自动允许同时发生时，以 Main 先处理的结果为准，后到结果忽略，Renderer 只显示 Main 确认的最终状态。请求、结果和截止时间由 Main 持有，Renderer 重载后通过 `RuntimeSnapshot` 恢复，不能重置倒计时或显示已过期按钮。窗口失焦或最小化时不发送系统通知。诊断日志只记录 Session ID、RPC 请求 ID、结果和处理时间，不记录完整命令、文件内容或审批 `title`。审批方式只在当前运行期间显示；重开 Session 后只恢复 Tool Call 的最终状态，不为“手动允许”或“自动允许”增加持久化数据。
11. MVP 的权限按钮只管理三种审批模式，不提供每个工具的 `allow`、`prompt`、`deny` 编辑器。单工具规则以后作为高级设置处理。

## 权限控件补充

12. 已有 Session 切换权限时，按钮立即显示目标权限并带转圈。启动失败并恢复原 Runtime 后，按钮恢复显示原权限。
13. Session 列表不显示权限标签。权限只在当前 Session 的输入区查看和修改，避免列表信息过多。
14. 没有创建或选中 Session 时，输入区仍显示禁用的`权限：全部允许`。新建 Session 同样默认使用`全部允许`。这条决定替换此前“新建 Session 默认标准”的规则。
15. 已有配置缺少权限或值无效时统一按`全部允许`处理，由 Main 补写 `yolo`。不再把这类配置回退为`标准`。
16. 权限选择框打开后，键盘焦点放在当前权限上。方向键切换候选项，Enter 确认，避免打开后误按 Enter 改变权限。
17. 用户再次选择当前权限时，只关闭选择框，不重启 Runtime、不显示转圈，也不重复写配置。
18. 新建且尚未启动 Runtime 的 Session 修改权限时立即保存。即使用户没有发送消息就关闭 Desktop，重新打开后也必须保留选择；Runtime 仍等到首条消息发送时才启动。
19. 新 Session 的权限配置写入失败时，界面恢复上一次已保存的值，并在输入框附近显示“权限保存失败”。此时不启动或重启 Runtime。
20. `严格`、`标准`和`全部允许`使用相同的普通按钮样式。`全部允许`不使用红色或橙色警告样式；只有切换中显示转圈，发生错误时使用错误样式。
21. 权限切换成功后不显示额外提示。转圈结束且按钮保留目标权限即表示成功；只有切换或保存失败时才显示错误。
